## Последний QA-регресс и исправление

Был найден срочный регресс:

- у части танцоров танцы показывали почти одинаковые результаты;
- судейские блоки скрывались.

Причины:

- `client_data.numeric_marks` отсутствовал в JSON, поэтому `filteredNumericMarks` в HTML был пустым;
- protocol-level итоговое место участника ошибочно раскладывалось как dance-level `final_place` на каждый танец.

Исправлено:

- `client_data.numeric_marks` возвращён в JSON;
- `tournament_dance_results` снова строится по реальным dance-level results;
- protocol-level итоговое место осталось в истории турниров, но больше не подменяет место каждого танца.

Контрольный результат для Груздевой Александры `2016461`:

```text

```

Средние по стандарту снова различаются:


2. Исключить дублирование marks при повторном парсинге.

4. Довести PDF export до реализованного состояния, если он всё ещё нужен как продуктовый слой.

## Основные ограничения

- Текущая база исторически строилась как Санкт-Петербург-only.
- Не-SPb протоколы требуют отдельного incremental import.
- Нельзя вручную добавлять города в отчёт без импортированного протокола.
- Нельзя подменять город турнира городом клуба/участника.
- Нельзя использовать protocol-level итоговое место как место каждого танца.
- Judge rankings в пользовательском отчёте показываются только при `n_marks >= 10` на судью.
