# Project Context

Документ восстанавливает базовый контекст проекта танцевальной аналитики, чтобы дальнейшая работа не зависела от истории чатов.

## Цель проекта

Проект строит аналитику спортивных бальных танцев по протоколам Compreg.

Источники данных:

- Compreg;
- HTML-протоколы;
- локальный cache протоколов;
- SQLite database.

Основные сущности:

- dancer;
- tournament;
- protocol;
- judge;
- marks;
- dance.

## Базовые правила аналитики

- Не делать домыслов.
- Использовать только данные протоколов.
- Любой вывод должен подтверждаться данными.
- В пользовательском тексте использовать названия `Стандарт` и `Латина`.
- Для solo entry использовать формулировку `солистка`, а не `пара`.
- Не смешивать программы, категории, города и типы оценок без явной отметки.
- Кресты, numeric marks, итоговые места и judge-level diagnostics должны оставаться разными слоями данных.

## Основная аналитика

Поддерживаемые направления анализа:

- динамика результатов;
- анализ танцев;
- анализ судей;
- строгие и мягкие судьи;
- согласованность панели;
- влияние судей;
- стабильность результатов;
- анализ скейтинга;
- турнирная статистика.

## Основной контрольный кейс

Основной тестовый танцор для проверки аналитики и отчётов:

```text
Груздева Александра
IDD: 2016461
```

Контрольный кейс используется для проверки:

- корректности dance-level metrics;
- блока `Сравнение танцев`;
- судейских блоков;
- фильтра периода;
- HTML/mobile layout;
- root HTML-файлов для GitHub Pages.

## Текущая архитектура

Данные:

- SQLite: `database/compreg_spb_2025_2026.sqlite`;
- cache протоколов: `data/cache/protocols/`;
- JSON промежуточного слоя: `reports/dancer_<idd>_report.json`.

Отчёты:

- HTML реализован;
- PDF export пока не найден в коде и считается будущим этапом.

Поддерживаются:

- отчёт танцора;
- trainer mode;
- аналитика турниров;
- мобильная версия;
- фильтр периода и городов в HTML-отчёте;
- root-копии HTML/JSON отчётов для GitHub Pages.

Основной pipeline:

```text
Compreg / cache
↓
SQLite
↓
build_dancer_report.py
↓
report JSON
↓
render_html_report.py
↓
HTML report
↓
root HTML/JSON for GitHub Pages
```

## GitHub Pages

GitHub Pages публикует статические HTML/JSON файлы из корня репозитория, а не Jinja-шаблоны.

После изменений в шаблоне, встроенном JavaScript, фильтрах, расчётах или отображении данных нужно:

1. Пересобрать отчёты.
2. Обновить root-копии `dancer_*_report.html/json`.
3. Проверить, что root HTML содержит актуальные маркеры изменений.
4. Закоммитить сгенерированные HTML/JSON, если они используются Pages.
5. Выполнить push.
6. Проверить опубликованную страницу GitHub Pages с hard refresh.

## Важное найденное ограничение

`scripts/build_spb_database.py` импортирует только турниры Санкт-Петербурга.

Следствие:

- не-SPb турниры могут находиться в cache;
- без отдельного incremental import они отсутствуют в SQLite;
- фильтр городов и отчёты будут показывать только те города, протоколы которых реально импортированы в SQLite.

## Направление дальнейшей разработки

Не выполнять ручной импорт отдельных танцоров как основной workflow.

Нужен общий incremental pipeline для каждого танцора:

- искать все протоколы в cache;
- сравнивать с SQLite;
- импортировать только отсутствующие `protocol_id`;
- не дублировать marks;
- пересобирать JSON и HTML;
- обновлять root-копии для GitHub Pages.
